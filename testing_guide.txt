SECTION 8 — ENDPOINT TESTS  (curl — PowerShell format)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

IMPORTANT — API body vs DB column names
  The API request/response uses friendly names. The data layer maps them to DB:
    API  "title"       →  DB  assmt_title
    API  "description" →  DB  assmt_brief
    API  "source_kind" →  DB  topic_source
    API  "rubric"      →  DB  rubric_relevance + rubric_blooms + rubric_depth
  So always use the API names in request bodies, not the DB column names.

RUBRIC FORMAT
  The rubric object in POST/PATCH bodies must have exactly these three keys
  and the values must sum to 10:
    {"relevance": 4, "blooms": 3, "depth": 3}   ✓  (4+3+3 = 10)
    {"relevance": 5, "blooms": 3, "depth": 2}   ✓  (5+3+2 = 10)
    {"correct": 1, "incorrect": -0.25}           ✗  (WRONG format — DB will reject)

RECIPIENTS FORMAT
  Each recipient item uses the key  "kind"  (not  "type"):
    [{"kind": "section", "id": 10}]              ✓
    [{"type": "section", "id": 10}]              ✗  (WRONG — ignored by data layer)

PowerShell note: if curl resolves to Invoke-WebRequest, use curl.exe instead.

8.3  POST /assessments — Create assessment
────────────────────────────────────────────────────────────────────────────────

BODY FIELD REFERENCE:
  title            — maps to DB assmt_title         (required)
  description      — maps to DB assmt_brief          (optional)
  source_kind      — "topic" or "document"          (required)
  topic_ids        — list of topic IDs, required when source_kind=topic
  document_id      — doc_id from ca_documents, required when source_kind=document
  subject_code     — used when source_kind=topic     (optional)
  recipients       — list of {kind: "section", id: N} items (required)
  question_count   — integer                         (required)
  duration_minutes — integer                         (required)
  rubric           — {relevance, blooms, depth} summing to 10  (required)
  start_time       — ISO 8601 string, e.g. "2026-06-12T09:00:00"  (optional)
  end_time         — ISO 8601 string                              (optional)
  status           — "draft" (default) or "live"    (optional)

--- Status transitions (allowed vs blocked) ---
  draft → draft, scheduled, live       ✓  allowed
  scheduled → draft, scheduled, live   ✓  allowed

SECTION 10 — COMMON ERRORS AND FIXES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Problem: 401 Unauthorized on every request
Fix: Add header  Authorization: Bearer 1  (the user id must exist in
     college_account_new.id).

Problem: Server crashes on startup with ModuleNotFoundError
Fix: pip install -r requirements.txt  (ensure venv is active).

Problem: "Can't connect to MySQL server"
Fix: Confirm MySQL is running and .env has correct DB_* values.
     Test:  mysql -u root -p edwisely_cm_db

Problem: 500 Failure on a specific endpoint
Fix: Check the Flask terminal — it logs the exception traceback.
     Common cause: missing table or column in the DB.
     Run the migration first if not done:
       mysql -u root -p edwisely_cm_db < production_migration.sql

Problem: 400 No Data Found on GET endpoints
Fix: The seed INSERTs in Section 4 need to be run first.
     Verify the tables have rows:
       SELECT * FROM curiosity_assessment;
       SELECT * FROM ca_has_students;

Problem: CHECK constraint fails on curiosity_assessment INSERT
Fix: rubric_relevance + rubric_blooms + rubric_depth must equal exactly 10.
     e.g.  4 + 3 + 3 = 10  ✓

Problem: KeyError: '<table_name>' on startup
Fix: metadata.reflect() couldn't find that table in the DB.
     Check: SHOW TABLES LIKE 'ca_%';
     Run the migration if tables are missing.

Problem: curl resolves to Invoke-WebRequest in PowerShell
Fix: Use curl.exe explicitly, or use Postman.
